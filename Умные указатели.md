## Умные указатели (Smart Pointers)

### Введение: проблема сырых указателей

До C++11 основным способом работы с динамической памятью были **сырые указатели** (raw pointers). Мы выделяли память с помощью `new` и освобождали с помощью `delete`. Это просто, но таит в себе множество опасностей.

Рассмотрим классический пример:

```cpp
#include <iostream>

class Resource {
public:
    Resource() { std::cout << "Resource acquired\n"; }
    ~Resource() { std::cout << "Resource released\n"; }
    void doSomething() { std::cout << "Doing something\n"; }
};

void problematicFunction() {
    Resource* ptr = new Resource();   // Выделили ресурс
    
    // ... много кода ...
    // Представьте, что здесь может возникнуть исключение или ранний return
    
    delete ptr;   // Освободили ресурс
}
```

**Какие проблемы здесь скрыты?**

1. **Утечка памяти (memory leak)**: Если между `new` и `delete` произойдет исключение или сработает `return`, то `delete` не выполнится, и память, занятая объектом `Resource`, никогда не будет освобождена.

2. **Висячие указатели (dangling pointers)**:
   ```cpp
   Resource* p = new Resource();
   delete p;
   p->doSomething();   // p указывает на уже освобожденную память! Неопределенное поведение.
   ```

3. **Двойное удаление (double delete)**:
   ```cpp
   Resource* p1 = new Resource();
   Resource* p2 = p1;
   delete p1;
   delete p2;   // Попытка удалить уже удаленный объект! Крах программы.
   ```

4. **Неясное владение (ownership ambiguity)**: Глядя на функцию, принимающую `Resource*`, непонятно: она берет объект во временное пользование или забирает его навсегда (и должна удалить)? Документация часто забывается, а код читать сложно.

Идея **умных указателей** проста: **связать время жизни динамически выделенного объекта с временем жизни объекта на стеке**, используя принцип **RAII** (Resource Acquisition Is Initialization — получение ресурса есть инициализация). Когда объект-указатель создается, он захватывает ресурс. Когда он выходит из области видимости (например, завершается функция), его деструктор автоматически освобождает ресурс. Это делает код безопасным относительно исключений и предотвращает утечки.

Стандартная библиотека C++11 предоставляет три основных типа умных указателей (в заголовочном файле `<memory>`):
- `std::unique_ptr` — единоличное владение.
- `std::shared_ptr` — совместное владение с подсчетом ссылок.
- `std::weak_ptr` — слабая ссылка для наблюдения за `shared_ptr` без продления жизни.

Давайте разберем каждый из них.

---

### 1. `std::unique_ptr` — уникальное владение

`std::unique_ptr` моделирует **исключительное владение** динамическим объектом. Это значит, что в любой момент времени только один `unique_ptr` может указывать на данный объект. Его нельзя скопировать, но можно **переместить** (с помощью `std::move`), передавая владение другому указателю. Когда `unique_ptr` уничтожается, он автоматически вызывает `delete` для своего объекта.

Это самый легковесный умный указатель: по размеру и производительности он идентичен сырому указателю (обычно ровно один машинный указатель), но при этом полностью безопасен.

#### Базовое использование

```cpp
#include <iostream>
#include <memory>   // для std::unique_ptr, std::make_unique

class Resource {
public:
    Resource() { std::cout << "Resource acquired\n"; }
    ~Resource() { std::cout << "Resource released\n"; }
    void doWork() { std::cout << "Working...\n"; }
};

int main() {
    // Создаем unique_ptr, владеющий объектом Resource (предпочтительный способ с C++14)
    std::unique_ptr<Resource> resPtr = std::make_unique<Resource>();
    
    // Используем как обычный указатель (благодаря перегруженным операторам * и ->)
    resPtr->doWork();
    (*resPtr).doWork();   // эквивалентно
    
    // Не нужно вызывать delete! Когда resPtr выйдет из области видимости,
    // деструктор unique_ptr автоматически освободит память.
    
    return 0;
}
```

**Вывод:**
```
Resource acquired
Working...
Working...
Resource released
```

#### Перемещение владения

Поскольку копирование запрещено, для передачи владения используется перемещение:

```cpp
void takeOwnership(std::unique_ptr<Resource> ptr) {
    std::cout << "Inside takeOwnership\n";
    ptr->doWork();
    // Здесь ptr уничтожается, освобождая ресурс
}

int main() {
    auto ptr1 = std::make_unique<Resource>();
    
    // takeOwnership(ptr1);   // ОШИБКА! unique_ptr нельзя копировать
    
    takeOwnership(std::move(ptr1));   // OK, владение передано
    
    if (!ptr1) {
        std::cout << "ptr1 теперь пуст\n";
    }
    
    return 0;
}
```

После перемещения `ptr1` становится `nullptr`. Это гарантирует, что ресурс всегда имеет ровно одного владельца.

#### Возврат из функции

`unique_ptr` очень удобно возвращать из функций, так как компилятор автоматически применяет семантику перемещения:

```cpp
std::unique_ptr<Resource> createResource() {
    return std::make_unique<Resource>();   // Происходит перемещение
}

int main() {
    auto ptr = createResource();   // Безопасно, владение переходит сюда
    ptr->doWork();
    return 0;
}
```

#### Работа с массивами

`std::unique_ptr` имеет специализацию для массивов:

```cpp
int main() {
    // Создаем массив из 10 целых чисел
    std::unique_ptr<int[]> arr = std::make_unique<int[]>(10);
    
    for (int i = 0; i < 10; ++i) {
        arr[i] = i * i;   // Доступ по индексу
        std::cout << arr[i] << " ";
    }
    std::cout << std::endl;
    
    // Деструктор unique_ptr вызовет delete[], а не delete
    return 0;
}
```

#### Как устроен `unique_ptr` внутри (упрощенно)

```cpp
template <typename T>
class unique_ptr {
private:
    T* ptr_;   // сырой указатель на управляемый объект
public:
    // Конструктор
    explicit unique_ptr(T* p = nullptr) : ptr_(p) {}
    
    // Запрет копирования
    unique_ptr(const unique_ptr&) = delete;
    unique_ptr& operator=(const unique_ptr&) = delete;
    
    // Перемещающий конструктор
    unique_ptr(unique_ptr&& other) noexcept : ptr_(other.ptr_) {
        other.ptr_ = nullptr;
    }
    
    // Перемещающее присваивание
    unique_ptr& operator=(unique_ptr&& other) noexcept {
        if (this != &other) {
            delete ptr_;                // Освобождаем старый ресурс
            ptr_ = other.ptr_;
            other.ptr_ = nullptr;
        }
        return *this;
    }
    
    // Деструктор
    ~unique_ptr() {
        delete ptr_;
    }
    
    // Доступ к объекту
    T* operator->() const { return ptr_; }
    T& operator*() const { return *ptr_; }
    
    // Получение сырого указателя (осторожно!)
    T* get() const { return ptr_; }
    
    // Освобождение владения без удаления
    T* release() {
        T* tmp = ptr_;
        ptr_ = nullptr;
        return tmp;
    }
    
    // Замена управляемого объекта
    void reset(T* p = nullptr) {
        delete ptr_;
        ptr_ = p;
    }
    
    // Проверка на пустоту
    explicit operator bool() const { return ptr_ != nullptr; }
};
```

**Ключевые моменты:**
- Размер `unique_ptr` равен размеру одного указателя (если не используется пользовательский deleter).
- Перемещение дешево (просто копирование указателя и зануление источника).
- При уничтожении всегда вызывается `delete` (или пользовательский deleter).

### 2. `std::shared_ptr` — совместное владение с подсчетом ссылок

`std::shared_ptr` предназначен для случаев, когда **несколько владельцев** должны разделять один и тот же динамический объект. Объект будет существовать до тех пор, пока хотя бы один `shared_ptr` на него ссылается. Это достигается с помощью **подсчета ссылок** (reference counting) — специального счетчика, который хранится в общей управляющей структуре.

#### Базовое использование

```cpp
#include <iostream>
#include <memory>

class Resource {
public:
    Resource() { std::cout << "Resource acquired\n"; }
    ~Resource() { std::cout << "Resource released\n"; }
    void doWork() { std::cout << "Working...\n"; }
};

int main() {
    // Создаем shared_ptr (предпочтительно с make_shared)
    std::shared_ptr<Resource> ptr1 = std::make_shared<Resource>();
    
    std::cout << "Use count: " << ptr1.use_count() << std::endl;   // 1
    
    {
        std::shared_ptr<Resource> ptr2 = ptr1;   // Копирование увеличивает счетчик
        std::cout << "Use count after copy: " << ptr1.use_count() << std::endl;   // 2
        
        ptr2->doWork();
    }   // ptr2 выходит из области видимости, счетчик уменьшается (но не до нуля)
    
    std::cout << "Use count after inner block: " << ptr1.use_count() << std::endl;   // 1
    
    ptr1->doWork();
    
    return 0;
}   // ptr1 уничтожается, счетчик становится 0, объект Resource удаляется
```

**Вывод:**
```
Resource acquired
Use count: 1
Use count after copy: 2
Working...
Use count after inner block: 1
Working...
Resource released
```

#### Как устроен `shared_ptr` внутри (упрощенно)

`shared_ptr` хранит два указателя:
1. Указатель на управляемый объект (как `unique_ptr`).
2. Указатель на **управляющий блок** (control block), который содержит:
   - Счетчик сильных ссылок (количество `shared_ptr`).
   - Счетчик слабых ссылок (количество `weak_ptr`).
   - Указатель на deleter (функцию удаления).
   - (Опционально) сам объект, если использовался `make_shared`.

```cpp
template <typename T>
class shared_ptr {
private:
    T* ptr_;                // Указатель на объект
    control_block* cb_;     // Указатель на управляющий блок
    
    struct control_block {
        size_t strong_refs;   // Количество shared_ptr
        size_t weak_refs;     // Количество weak_ptr
        // Deleter и аллокатор (упрощенно опущены)
        
        control_block() : strong_refs(1), weak_refs(0) {}
    };
    
public:
    // Конструктор из сырого указателя
    explicit shared_ptr(T* p) : ptr_(p), cb_(new control_block()) {}
    
    // Копирующий конструктор
    shared_ptr(const shared_ptr& other) : ptr_(other.ptr_), cb_(other.cb_) {
        ++cb_->strong_refs;
    }
    
    // Деструктор
    ~shared_ptr() {
        if (--cb_->strong_refs == 0) {
            delete ptr_;          // Удаляем объект
            if (cb_->weak_refs == 0) {
                delete cb_;       // Удаляем управляющий блок, если нет слабых ссылок
            }
        }
    }
    
    // ... операторы присваивания, доступ и т.д.
    
    size_t use_count() const { return cb_->strong_refs; }
};
```

**Важно:** Создание `shared_ptr` через сырой указатель **должно происходить только один раз** для данного объекта. Иначе будут созданы независимые управляющие блоки, и объект будет удален несколько раз!

```cpp
Resource* raw = new Resource();
std::shared_ptr<Resource> sp1(raw);      // OK
std::shared_ptr<Resource> sp2(raw);      // ОШИБКА! Второй управляющий блок, двойное удаление
```

**Решение:** Всегда используйте `std::make_shared` или создавайте последующие `shared_ptr` только копированием существующего.

#### `std::make_shared` против `new`

`std::make_shared<T>(args...)` создает объект типа `T` и `shared_ptr` на него за одну операцию выделения памяти. При использовании `new` происходят два отдельных выделения: для объекта и для управляющего блока. `make_shared` более эффективен по памяти и производительности, а также безопаснее с точки зрения исключений.

```cpp
// Предпочтительно:
auto sp = std::make_shared<Resource>();

// Менее эффективно (два выделения памяти):
std::shared_ptr<Resource> sp2(new Resource());
```

#### Потенциальные проблемы `shared_ptr`

1. **Циклические ссылки (circular references)** — самая серьезная ловушка. Если два объекта владеют `shared_ptr` друг на друга, их счетчики ссылок никогда не станут нулевыми, и память не освободится. Решение — `std::weak_ptr` (см. следующий раздел).

2. **Накладные расходы**: `shared_ptr` тяжелее `unique_ptr` как по памяти (два указателя вместо одного), так и по времени (атомарные операции со счетчиком для потокобезопасности).

3. **Ложное чувство безопасности**: `shared_ptr` не делает код потокобезопасным автоматически. Он гарантирует, что объект не будет удален, пока вы его используете, но сам объект может требовать синхронизации доступа.

### 3. `std::weak_ptr` — слабая ссылка

`std::weak_ptr` — это "наблюдатель" за объектом, управляемым `shared_ptr`. Он **не увеличивает счетчик сильных ссылок**, поэтому не продлевает жизнь объекту. Основное назначение — разрыв циклических зависимостей.

#### Базовое использование

```cpp
#include <iostream>
#include <memory>

int main() {
    std::weak_ptr<int> wp;
    
    {
        auto sp = std::make_shared<int>(42);
        wp = sp;   // wp теперь наблюдает за sp
        
        // Чтобы получить доступ к объекту, нужно "заблокировать" weak_ptr
        if (std::shared_ptr<int> locked = wp.lock()) {
            std::cout << "Value: " << *locked << ", use_count: " << locked.use_count() << std::endl;
            // use_count будет 2 (sp и locked)
        }
        
        std::cout << "wp.expired(): " << wp.expired() << std::endl;   // false
    }   // sp уничтожен, объект удален
    
    std::cout << "wp.expired(): " << wp.expired() << std::endl;   // true
    
    // Попытка заблокировать просроченный weak_ptr вернет пустой shared_ptr
    if (auto locked = wp.lock()) {
        // Этот блок не выполнится
    } else {
        std::cout << "Object is gone\n";
    }
    
    return 0;
}
```

#### Решение проблемы циклических ссылок

Рассмотрим классический пример: двусвязный список, где каждый узел содержит указатели на предыдущий и следующий. Если использовать `shared_ptr`, узлы никогда не удалятся.

```cpp
#include <memory>
#include <iostream>

class Node {
public:
    std::shared_ptr<Node> next;
    std::shared_ptr<Node> prev;
    int value;
    
    Node(int v) : value(v) {
        std::cout << "Node " << value << " created\n";
    }
    ~Node() {
        std::cout << "Node " << value << " destroyed\n";
    }
};

int main() {
    auto node1 = std::make_shared<Node>(1);
    auto node2 = std::make_shared<Node>(2);
    
    node1->next = node2;
    node2->prev = node1;
    
    // node1 и node2 выйдут из области видимости, но их деструкторы НЕ вызовутся!
    // Потому что они ссылаются друг на друга.
    return 0;
}
```

**Вывод (только создание, без уничтожения):**
```
Node 1 created
Node 2 created
```

Чтобы исправить, одну из связей нужно сделать слабой:

```cpp
class Node {
public:
    std::shared_ptr<Node> next;
    std::weak_ptr<Node> prev;   // Слабая ссылка на предыдущий узел
    // ...
};

int main() {
    auto node1 = std::make_shared<Node>(1);
    auto node2 = std::make_shared<Node>(2);
    
    node1->next = node2;
    node2->prev = node1;
    
    // Теперь при выходе из main:
    // node1 уничтожается (счетчик 1->0)
    // node2 уничтожается (счетчик 1->0, так как node1 уже нет)
    return 0;
}
```

**Вывод:**
```
Node 1 created
Node 2 created
Node 1 destroyed
Node 2 destroyed
```

#### Как работает управляющий блок с `weak_ptr`

Управляющий блок хранит два счетчика:
- **strong_refs** — количество `shared_ptr`.
- **weak_refs** — количество `weak_ptr`.

Объект удаляется, когда `strong_refs == 0`. Сам управляющий блок удаляется, когда **оба счетчика равны нулю**. Это позволяет `weak_ptr` проверять, жив ли еще объект, обращаясь к управляющему блоку, даже если самого объекта уже нет.

### 4. Сравнение умных указателей и выбор подходящего

| Критерий | `std::unique_ptr` | `std::shared_ptr` | `std::weak_ptr` |
| :--- | :--- | :--- | :--- |
| **Владение** | Единоличное | Совместное | Не владеет (наблюдает) |
| **Копирование** | Запрещено | Разрешено (увеличивает счетчик) | Разрешено |
| **Перемещение** | Разрешено | Разрешено | Разрешено |
| **Накладные расходы** | Минимальные (как сырой указатель) | Два указателя + атомарный счетчик | Как `shared_ptr` |
| **Использование** | По умолчанию для динамических объектов | Когда нужно разделять владение | Для разрыва циклов, кэширования |

**Золотое правило:** Если вы не уверены, какой указатель использовать, начинайте с `std::unique_ptr`. Если позже окажется, что нужно разделяемое владение, вы всегда сможете преобразовать `unique_ptr` в `shared_ptr` (но не наоборот). `shared_ptr` вводит накладные расходы и сложность, поэтому используйте его только когда действительно необходимо совместное владение.

### 5. Умные указатели и наследование

Умные указатели корректно работают с полиморфизмом. Можно хранить `unique_ptr<Derived>` в `unique_ptr<Base>` при перемещении:

```cpp
#include <iostream>
#include <memory>
#include <vector>

class Animal {
public:
    virtual ~Animal() = default;
    virtual void speak() const = 0;
};

class Dog : public Animal {
public:
    void speak() const override { std::cout << "Woof!\n"; }
};

class Cat : public Animal {
public:
    void speak() const override { std::cout << "Meow!\n"; }
};

int main() {
    std::vector<std::unique_ptr<Animal>> animals;
    
    animals.push_back(std::make_unique<Dog>());
    animals.push_back(std::make_unique<Cat>());
    
    for (const auto& animal : animals) {
        animal->speak();
    }
    
    // Вектор сам очистит память при разрушении
    return 0;
}
```

**Важно:** Базовый класс **должен иметь виртуальный деструктор**, иначе при удалении через указатель на базу будет вызван только деструктор базы, что приведет к утечкам (как и с сырыми указателями). `unique_ptr` и `shared_ptr` не спасают от этой проблемы — они вызывают `delete` для того типа, который хранится в указателе. Если в `unique_ptr<Base>` хранится `Derived`, а деструктор `Base` не виртуален, то `delete` вызовет только `~Base()`.

### 6. Практические примеры из реальной разработки

#### Фабрика, возвращающая `unique_ptr`

```cpp
#include <memory>
#include <string>

class Product {
public:
    virtual ~Product() = default;
    virtual std::string getName() const = 0;
};

class ConcreteProductA : public Product {
public:
    std::string getName() const override { return "A"; }
};

class ConcreteProductB : public Product {
public:
    std::string getName() const override { return "B"; }
};

class Factory {
public:
    static std::unique_ptr<Product> createProduct(char type) {
        switch (type) {
            case 'A': return std::make_unique<ConcreteProductA>();
            case 'B': return std::make_unique<ConcreteProductB>();
            default:  return nullptr;
        }
    }
};

int main() {
    auto product = Factory::createProduct('A');
    if (product) {
        // Используем продукт
    }
    // Память освободится автоматически
    return 0;
}
```

#### Кэширование с `weak_ptr`

```cpp
#include <iostream>
#include <memory>
#include <unordered_map>
#include <string>

class Texture {
public:
    Texture(const std::string& name) : name_(name) {
        std::cout << "Loading texture: " << name << std::endl;
    }
    ~Texture() {
        std::cout << "Unloading texture: " << name_ << std::endl;
    }
    void draw() const { std::cout << "Drawing " << name_ << std::endl; }
private:
    std::string name_;
};

class TextureCache {
private:
    std::unordered_map<std::string, std::weak_ptr<Texture>> cache_;
public:
    std::shared_ptr<Texture> getTexture(const std::string& name) {
        auto it = cache_.find(name);
        if (it != cache_.end()) {
            // Пытаемся получить существующую текстуру
            if (auto tex = it->second.lock()) {
                std::cout << "Cache hit for " << name << std::endl;
                return tex;
            }
        }
        // Текстуры нет или она была удалена, загружаем заново
        std::cout << "Cache miss for " << name << std::endl;
        auto tex = std::make_shared<Texture>(name);
        cache_[name] = tex;   // Сохраняем слабую ссылку
        return tex;
    }
};

int main() {
    TextureCache cache;
    
    {
        auto tex1 = cache.getTexture("player.png");
        tex1->draw();
        
        auto tex2 = cache.getTexture("player.png");   // Попадание в кэш
        tex2->draw();
    }   // tex1 и tex2 уничтожены, текстура "player.png" выгружается
    
    // Позже снова запрашиваем ту же текстуру
    auto tex3 = cache.getTexture("player.png");   // Промах кэша, загрузка заново
    tex3->draw();
    
    return 0;
}
```

**Вывод:**
```
Cache miss for player.png
Loading texture: player.png
Drawing player.png
Cache hit for player.png
Drawing player.png
Unloading texture: player.png
Cache miss for player.png
Loading texture: player.png
Drawing player.png
Unloading texture: player.png
```

Этот пример демонстрирует, как `weak_ptr` позволяет кэшировать объекты, не мешая им удаляться, когда они больше нигде не используются.

### 7. Чего не делают умные указатели

1. **Не решают проблему висячих ссылок полностью**: Если вы сохраните сырой указатель, полученный через `.get()`, и `shared_ptr`/`unique_ptr` будет уничтожен, сырой указатель станет висячим.

   ```cpp
   int* raw = nullptr;
   {
       auto sp = std::make_shared<int>(10);
       raw = sp.get();
   }
   // raw теперь висячий! Не используйте его.
   ```

2. **Не обеспечивают потокобезопасность объекта**: Если несколько потоков имеют `shared_ptr` на один объект и модифицируют его, необходима дополнительная синхронизация.

3. **Не отменяют необходимость виртуальных деструкторов**: Как уже упоминалось, деструктор базового класса должен быть виртуальным.

### Заключение

Умные указатели — это фундаментальный инструмент современного C++. Они превращают ручное управление памятью из источника ошибок и утечек в надежную и выразительную часть языка.

**Краткая памятка:**
- По умолчанию используйте `std::unique_ptr`.
- Если нужно разделяемое владение, используйте `std::shared_ptr`, но помните о циклических ссылках.
- Для разрыва циклов и наблюдения используйте `std::weak_ptr`.
- Создавайте указатели через `std::make_unique` (C++14) и `std::make_shared`.
- Никогда не используйте сырые `new` и `delete` напрямую, если можно обойтись умными указателями.
